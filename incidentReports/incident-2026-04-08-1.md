# Incident: 2026-04-08 

->Summary

During the scheduled chaos testing period on **2026-04-08**, the JWT Pizza service experienced multiple waves of instability throughout the day. The main symptoms were repeated authentication failures, elevated backend error rates, increased request latency, temporary monitoring gaps, and intermittent instance health issues.
Note: InstanceDown alerts observed in the system were part of earlier manual testing and are not included in this incident analysis.

The incident was detected through Grafana alerting and IRM notifications. The primary alerts triggered during the testing period were:

- High Latency Alert
- Auth Error Spike Alert
- General Error Spike Alert
- DatasourceNoData
- InstanceDown

This was a high severity incident simulation because the service showed degraded behavior across several observability dimensions at once: latency, authentication failures, backend errors, monitoring availability, and instance health. The incident was acknowledged and resolved through Grafana IRM after investigation confirmed the cause and behavior of the system under stress.


->Detection

The incident was detected through **automated Grafana alerting** and confirmed through dashboards and logs.

Alerts observed during the day included:

- **High Latency Alert**
- **Auth Error Spike Alert**
- **General Error Spike Alert**
- **DatasourceNoData**
- **InstanceDown**

The alerting system successfully paged the on call responder through the configured Grafana IRM escalation chain. Notifications were received by dashboard, mobile push, email, and SMS during the event.

The most important detection signals were:

- rising authentication failures on the authentication panel
- repeated backend error logs containing `unknown user`
- elevated request latency, which rose to roughly **500 ms**
- repeated firing and resolution cycles in Grafana IRM alert groups
- temporary datasource and instance health alerts

Time to detection was good because the alert rules were already configured against the service’s logs and metrics. A future improvement would be tightening the distinction between general backend errors and authentication specific spikes so that related alerts are easier to correlate.


->Impact

During the chaos window, the service remained partially operational, but it experienced degraded reliability and performance.

Observed impact included:

- large spikes in failed authentication attempts
- repeated backend exceptions on `/api/auth`
- sustained latency increase to around **500 ms**
- intermittent drops and spikes in request behavior
- temporary monitoring disruption shown by `DatasourceNoData`
- instance health alerts shown by `InstanceDown`

Internal observability was also affected because monitoring temporarily lost access to some expected data sources during parts of the incident.

The system did not appear completely unavailable for the entire period, but users would likely have experienced slower responses, failed authentication attempts, and intermittent instability.

->Response

After receiving alerts from Grafana IRM, I investigated the incident using both metrics and logs dashboards.

The response steps included:

1. Reviewing Grafana alert groups and confirming which alerts were firing.
2. Checking metrics dashboards for:
   - authentication spikes
   - latency increases
   - request rate changes
   - CPU and memory behavior
3. Checking logs dashboards for:
   - repeated `/api/auth` failures
   - `unknown user` exceptions
   - related backend request and SQL activity
4. Validating that alert notifications were functioning correctly through mobile, email, and SMS delivery.
5. Creating and refining alerts to ensure the failure patterns were being detected properly.
6. Acknowledging and resolving alert groups after confirming the observed behavior and allowing the chaos wave to subside.


Root cause

The root cause of the incident was intentional chaos injection combined with simulated traffic, especially repeated invalid authentication attempts to `/api/auth`.

The most direct contributing factors were:

- repeated invalid login attempts using non existent users
- backend exception generation for `unknown user`
- increased request load on the authentication path
- resulting performance degradation, including latency increase
- temporary instability in observability and infrastructure signals, reflected in `DatasourceNoData` and `InstanceDown`

This was not a single code bug that needed to be fixed in the service source. Instead, the chaos testing process revealed that the authentication path and backend error handling were key weak points under stress.


->Resolution

The incident was resolved operationally by:

- investigating the alert waves through Grafana dashboards and logs
- confirming the cause of the alerts
- stopping or reducing the test traffic when needed
- allowing the service to recover after chaos conditions subsided
- acknowledging and resolving the alert groups in Grafana IRM

The service was considered recovered once:

- active alert groups stopped firing
- dashboards showed stable resource usage
- latency returned toward a steady range
- error spike alerts were no longer active
- alert groups were marked as resolved in Grafana IRM

The response process validated that the alerting pipeline, escalation chain, and dashboards were all functioning as intended.


->Prevention

The incident shows several areas where the system could be improved to reduce the impact of similar real world failures.

Possible prevention improvements include:

- rate limiting repeated invalid authentication attempts
- better handling of invalid users to reduce repeated exception noise
- separating authentication specific alerts from broader backend error alerts more clearly
- improving datasource reliability so monitoring gaps are less likely during instability
- adding more precise thresholds or ratios for authentication failures versus normal traffic
- strengthening instance health monitoring correlation with service behavior

This incident also showed the value of maintaining both metrics based and log based alerts. Some failures were most visible through latency and traffic patterns, while others were most obvious in the logs.


->Action items

1. Add or refine rate limiting on `/api/auth` to reduce repeated invalid authentication abuse.
2. Improve backend handling of invalid users to avoid excessive repeated exception logs.
3. Tune authentication and general error alerts to reduce redundancy while keeping coverage.
4. Improve observability around datasource health so that `DatasourceNoData` is easier to investigate.
5. Review `InstanceDown` events alongside service metrics to determine whether infrastructure instability contributed to chaos impact.
6. Document the final alert set and expected chaos symptoms for future testing runs.
7. Keep both metrics and logs dashboards available during future chaos windows so incident evidence can be captured immediately.
